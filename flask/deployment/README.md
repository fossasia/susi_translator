# Deployment Architecture & Automated SSL

This directory contains the infrastructure configuration for deploying the Susi Translator Flask application. The architecture utilizes **Nginx** as a reverse proxy and **Certbot** for automated Let's Encrypt SSL certificate provisioning and renewal.

## Architecture Overview

To ensure zero-downtime deployments and handle the dependency loop between Nginx and Certbot (Nginx requires certificates to start, and Certbot requires domain validation to generate certificates), the system utilizes a **dummy certificate bootstrapping** mechanism alongside **DNS-01 challenges** via DuckDNS.

## Components

### Nginx Reverse Proxy (`/nginx`)

The Nginx container handles request routing, static file serving, and SSL termination.

- **`Dockerfile.nginx`**: A custom Alpine-based Nginx image that installs required dependencies (`openssl`) and provisions the initialization scripts.
- **`nginx.conf`**: The primary configuration file. It enforces HTTPS redirection, serves static assets directly, and proxies API traffic to the Flask backend. It is explicitly configured to handle long-running connections (Server-Sent Events and model warmup) without buffering.
- **`bootstrap_ssl.sh`**: An entrypoint script that executes before the Nginx daemon starts. It checks the shared Docker volume for existing Let's Encrypt certificates. If none are present, it generates a temporary self-signed certificate, allowing Nginx to boot successfully. It also initializes a background process that reloads the Nginx configuration every 6 hours to seamlessly apply any renewed certificates.

### Certbot Automated SSL (`/certbot`)

The Certbot container is responsible for the automated retrieval and renewal of Let's Encrypt certificates.

- **`entrypoint.sh`**: The main process for the Certbot container. It runs an infinite loop, checking the certificate status every 12 hours. If renewal is required, it triggers Certbot with the DNS-01 challenge configuration.
- **`authenticator.sh`**: A custom manual authentication hook for Certbot. Since DuckDNS is used for domain routing, this script integrates with the DuckDNS HTTP API to automatically inject the Let's Encrypt validation token into the domain's DNS `TXT` records. It enforces a 30-second delay to ensure global DNS propagation before Certbot attempts verification.

Certificates generated by Certbot are stored in the `certbot_certs` Docker volume, which is shared directly with the Nginx container.

## Deployment Instructions

1. **Environment Configuration**
   Ensure the AWS Elastic IP is mapped to the DuckDNS domain, and the root `.env` file is populated with:

   - `DOMAIN_NAME` (e.g., `susi.duckdns.org`)
   - `DUCKDNS_TOKEN` (From the DuckDNS dashboard)
   - `CERTBOT_EMAIL` (For Let's Encrypt expiration notices)

2. **Initialize the Stack**
   Execute the following command from the `flask/` directory:

   ```bash
   docker-compose up -d --build
   ```

3. **Finalize SSL Configuration**
   During the initial startup, Nginx will boot using a temporary self-signed certificate while Certbot processes the DNS challenge. Monitor Certbot's progress using `docker logs susi_certbot`. Once the real certificate is successfully retrieved, restart Nginx to apply it:

   ```bash
   docker restart susi_nginx
   ```

Following the initial deployment, the entire system is fully autonomous. Certbot will handle bi-monthly certificate renewals, and Nginx will automatically reload the renewed certificates into memory.
